# ZRead-generated project Wiki

The project Wiki is generated by the project-local `zread_cli` dependency from a clean isolated clone. `.zread/` is the complete project boundary: `config/index.yaml` composes ordered, secret-free YAML fragments, `scripts/` owns generation and validation, and `wiki/` contains the committed output. The adapter merges the fragments, injects the existing Kimi key only into a mode-600 config in an isolated temporary HOME, and executes `node_modules/.bin/zread` rather than a global binary. It publishes only the validated active version from `.zread/wiki/current` and normalizes vendor page metadata into a canonical committed manifest before Web consumes it. Git history replaces ZRead's local accumulation of old versions, so each generated documentation pull request contains one reviewable current snapshot.

The generation adapter extracts ZRead line citations into a canonical `sources.json` beside `wiki.json`. It verifies that every indexed path is an explicitly cited, regular text file inside the repository, rejects traversal and credential-bearing file names, validates each start line, and clamps only an overlong end line to the actual file length. The Next.js `/docs` route consumes those two indexes and the referenced Markdown at build time, emitting static `/docs/source/*` pages whose links target the canonical start-line anchor. A deployed Web artifact can therefore open cited code without reaching GitHub, rescanning generated Markdown, or exposing arbitrary runtime filesystem access. This keeps the Wiki inside the existing Web deployment and avoids adding `zread browse` as another production process. Generation fails when the CLI version, JSONL terminal event, write boundary, manifest, page files, source index, path safety, start line, or Markdown quality checks are invalid.

The generator uses ZRead's OpenAI-compatible provider interface. Language, concurrency, model and Kimi endpoint defaults are versioned in separate project config fragments; later fragments override earlier fields. Existing Kimi Coding environment names may override those defaults, while credentials are never accepted in committed fragments or copied into the child environment, command arguments, generated output, or logs. Generation remains an explicit local command or manually triggered GitHub workflow because it consumes external model quota and sends repository-derived content to the configured provider.

The ZRead CLI distribution is currently marked `UNLICENSED`, and long-running Kimi use remains subject to the provider's applicable plan and terms. The workflow has no schedule: manually dispatching it is the project owner's explicit operational acceptance of those external constraints for that run; this ADR does not replace a legal or procurement review.

We rejected deploying `zread browse` because it would duplicate the Web documentation surface and introduce another service lifecycle. We also rejected linking source citations to a remote code host because deployed project documentation must remain self-contained, rejected committing every ZRead historical version because it would make generated pull requests grow indefinitely, and rejected reading arbitrary repository or `.zread/` files because the active Wiki citations and `wiki.json` already define smaller auditable contracts.
