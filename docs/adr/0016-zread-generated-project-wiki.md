# ZRead-generated project Wiki

The project Wiki is generated by the project-local `zread_cli` dependency from a clean isolated clone. `.zread/` is the complete project boundary: `config/index.yaml` composes ordered, secret-free YAML fragments, `scripts/` owns generation and validation, and `wiki/` contains the committed output. The adapter merges the fragments, injects the existing Kimi key only into a mode-600 config in an isolated temporary HOME, and executes `node_modules/.bin/zread` rather than a global binary. It publishes only the validated active version from `.zread/wiki/current`. Git history replaces ZRead's local accumulation of old versions, so each generated documentation pull request contains one reviewable current snapshot.

The Next.js `/docs` route reads the active version's `wiki.json` as the navigation and routing source of truth, then renders the referenced Markdown at build time. This keeps the Wiki inside the existing Web deployment and avoids adding `zread browse` as another production process. Manifest paths and route slugs are validated before reading files, and generation fails when the CLI version, JSONL terminal event, write boundary, manifest, page files, or Markdown quality checks are invalid.

The generator uses ZRead's OpenAI-compatible provider interface. Language, concurrency, model and Kimi endpoint defaults are versioned in separate project config fragments; later fragments override earlier fields. Existing Kimi Coding environment names may override those defaults, while credentials are never accepted in committed fragments or copied into the child environment, command arguments, generated output, or logs. Generation remains an explicit local command or manually triggered GitHub workflow because it consumes external model quota and sends repository-derived content to the configured provider.

The ZRead CLI distribution is currently marked `UNLICENSED`, and long-running Kimi use remains subject to the provider's applicable plan and terms. The workflow has no schedule: manually dispatching it is the project owner's explicit operational acceptance of those external constraints for that run; this ADR does not replace a legal or procurement review.

We rejected deploying `zread browse` because it would duplicate the Web documentation surface and introduce another service lifecycle. We also rejected committing every ZRead historical version because it would make generated pull requests grow indefinitely, and rejected reading arbitrary files under `.zread/` because `wiki.json` already defines a smaller auditable contract.
