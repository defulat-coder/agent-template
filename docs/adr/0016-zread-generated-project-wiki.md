# ZRead-generated project Wiki

The project Wiki is generated by the project-local `zread_cli` dependency from a clean isolated clone. `.zread/` is the complete project boundary: `config/index.yaml` composes ordered, secret-free YAML fragments, `scripts/` owns generation and validation, and `wiki/` contains the committed output. The adapter merges the fragments, injects the existing Kimi key only into a mode-600 config in an isolated temporary HOME, and executes `node_modules/.bin/zread` rather than a global binary. After validating the active pointer, manifest, generated pages, cited source files and credential boundaries, it publishes ZRead's complete native Wiki directory without adding, deleting, normalizing or rewriting generated files. ZRead therefore remains the single source of truth for its on-disk layout.

The Next.js `/docs` route follows `current`, consumes the native `wiki.json`, and reads only Markdown pages listed by that manifest. At build time it extracts line citations from that closed page set to derive a minimal source allowlist and emits static `/docs/source/*` pages whose links target the cited start-line anchor. It rejects traversal, credential-bearing names, non-text files and paths not explicitly cited by the active Wiki. No project-owned index is written into the native version directory. A deployed Web artifact can therefore open cited code without reaching a remote code host or exposing arbitrary runtime filesystem access.

The Web documentation surface uses the shared shadcn/typeset stylesheet for Markdown semantics and shadcn primitives for navigation, responsive disclosure, breadcrumbs and source-file presentation. Mermaid fenced code is dynamically loaded in the browser and rendered with Mermaid's strict security level; ordinary fenced code remains source text. This keeps the Wiki inside the existing Web deployment and avoids adding `zread browse` as another production process.

The generator uses ZRead's OpenAI-compatible provider interface. Language, concurrency, model and Kimi endpoint defaults are versioned in separate project config fragments; later fragments override earlier fields. Existing Kimi Coding environment names may override those defaults, while credentials are never accepted in committed fragments or copied into the child environment, command arguments, generated output, or logs. Generation remains an explicit local command or manually triggered GitHub workflow because it consumes external model quota and sends repository-derived content to the configured provider.

The ZRead CLI distribution is currently marked `UNLICENSED`, and long-running Kimi use remains subject to the provider's applicable plan and terms. The workflow has no schedule: manually dispatching it is the project owner's explicit operational acceptance of those external constraints for that run; this ADR does not replace a legal or procurement review.

We rejected deploying `zread browse` because it would duplicate the Web documentation surface and introduce another service lifecycle. We also rejected linking source citations to a remote code host because deployed project documentation must remain self-contained, rejected project-owned manifest normalization and `sources.json` injection because they mutate the native ZRead directory, and rejected reading arbitrary repository or `.zread/` files because the active manifest and its citations already define smaller auditable contracts.
